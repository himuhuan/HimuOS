# Codex MVP 实施计划

## 核心判断

这轮目标不再定义为“完整用户子系统”，而定义为：

**一个正式成立、可论证、可演示的双进程用户态原型操作系统。**

它的判定标准不是功能铺开多少，而是最小架构闭环是否成立：

1. 用户程序必须是**编译型 C 程序**，而不是手写机器码。
2. 用户程序必须运行在**各自独立的 ExProcess / private root** 中。
3. 用户态必须通过现有 `int 0x80` 路径获取内核服务。
4. 两个用户进程必须能在调度器下共同运行；最低线可先依赖当前 RR，增强线再补入简化优先级。
5. 两个进程都必须完成自己的 teardown / terminate / reap 闭环。

基于这个目标，优先级调度从“明确不做”调整为**推荐并行增强项**。也就是说，它不再是阻塞 `P1/P2` 成立的硬前置，但团队既然还有余力，就值得做一个**简化版优先级调度**，用来增强展示效果并部分兑现 README 承诺。

与此同时，**键盘输入与 Shell 继续明确排除在本轮范围外**。这两者会显著拉长设备层和交互层战线，不利于当前原型按时收口。

---

## MVP 边界

### 必须保留

- 编译型用户程序工具链
- 双进程启动与运行
- 每进程独立地址空间
- capability seed block
- 最小可用 syscall 面
- 双进程退出与回收证据链
- 单命令可复现的 demo profile

### 推荐并行增强

- 简化版优先级调度
- 用户侧最小 ABI 包装层
- 将优先级能力接入最终主演示 profile

### 明确不做

- 键盘输入
- Shell / 用户态命令解释器
- 通用 ELF loader
- 动态 per-process layout allocator
- 完整 Object Manager
- 跨进程 handle / open / dup
- 用户 fault 可恢复
- 文件系统、SMP

## 学术约束口径：用户态只能通过 EX 访问 KE

### 约束定义

本项目在最终答辩与论文口径中，需要满足如下学术约束：

> **用户态不能直接访问 KE 的业务语义；用户态只能通过 EX 暴露的对象、句柄与系统调用 contract 请求服务。**

这里的“通过 EX 访问 KE”指的是**逻辑服务路径**，而不是要求第一条陷入后的机器指令必须位于 EX。  
`int 0x80` trap entry、中断帧保存/恢复、上下文切换、页表切换等内容天然属于 KE / arch 纯机制层，它们留在 KE 中是合理且必要的，并**不构成对该约束的违背**。

### 最终分层口径

为满足上述约束，最终设计必须按以下方式对外表述：

- `User` 只看见 `libsys.h`、seed block、handle、返回值约定
- `EX` 负责定义用户可见 syscall contract，并持有进程、线程、stdout service、waitable object、handle table 等策略语义
- `KE` 只提供 trap 进入、copy-in/out、调度、等待、地址空间切换、控制台底层输出等纯机制能力

换句话说，用户态向内核请求服务的正式路径应被描述为：

```text
User -> int 0x80 -> KE trap entry -> EX syscall dispatch/policy -> KE mechanism helpers
```

而不应被描述为：

```text
User -> KE raw syscall service
```

### 为何该约束在本方案中可以成立

本计划的主线是“编译型用户程序 + 双进程 + 最小 ABI + 可选简化优先级调度”。这一主线天然有利于满足学术约束：

- 双进程原型本身就是围绕 `ExProcess` / `ExThread` 展开的，而不是围绕裸 `KTHREAD` 展开
- 句柄、seed block、stdout、waitable object 这些用户态接触到的资源，本来就更适合由 EX 统一持有与解释
- 简化优先级调度属于 KE 机制增强，不会削弱“用户只能先经过 EX”这一逻辑边界
- 键盘与 Shell 暂不纳入本轮，反而降低了用户态接口在设备/交互层过早外溢的风险

### 设计收口条件

如果团队希望在最终答辩中明确声称“用户态只能通过 EX 访问 KE”，则本计划默认需要满足以下收口条件：

1. **最终用户可见 ABI 不再暴露 `SYS_RAW_*`。**  
   `SYS_RAW_WRITE`、`SYS_RAW_EXIT` 一类接口可以继续作为 bootstrap / internal path 存在，但不能再成为正式用户 contract 的一部分。

2. **所有正式用户 syscall 先进入 EX，再由 EX 调用 KE。**  
   也就是说，trap handler 可位于 KE，但正式 syscall number 的语义解释、对象解析、句柄校验、权限判断、退出语义都应由 EX 拥有。

3. **用户程序只通过 `libsys.h` 与公开 seed layout 工作。**  
   用户程序源码中不应直接写 raw trap number、raw syscall number 或依赖 KE 内部实现细节。

4. **主演示 profile 只使用 EX-facing 接口。**  
   最终双进程 demo 必须证明：两个进程都是通过 EX 暴露的 syscall / handle contract 获得服务，而不是借由 KE raw path 直接完成用户 I/O 或退出。

### 允许保留的内部实现

以下内容即使继续存在于 KE，也不影响上述学术约束成立：

- `int 0x80` 中断向量与 trap 入口
- 中断帧解析与寄存器搬运
- `copy-in/out`、页表查询、地址空间切换
- 调度、等待、线程退出等底层机制
- 供 EX 调用的底层 console / timing / synchronization helpers

也就是说，**约束针对的是“用户可见服务接口归谁定义与控制”，不是针对“硬件陷入代码放在哪个目录”。**

### 正式答辩表述

在满足上述收口条件后，本项目可以使用如下正式表述：

> HimuOS 采用 Ke/Ex 分层宏内核设计。用户态不能直接调用 KE 的业务接口，而只能通过 EX 暴露的对象化系统调用 contract 请求服务。KE 负责中断陷入、调度、地址空间切换与数据复制等纯机制；EX 负责进程/线程包装、句柄解析、权限校验、对象生命周期和系统调用语义。因而从学术意义上，用户态对内核服务的访问路径被严格约束为“User -> EX -> KE”。

## P1：编译型用户程序替换手写字节码

### 方向

先把“用户态存在”的证据，从手写字节数组升级成真正的 C 用户程序。这是整个 MVP 最关键的质变点，也是后续双进程的基础。

### 目标

- 引入最小 `src/user/` 目录
- 提供 `crt0.S`、`libsys.h`、`user.ld`
- 至少产出 1 个 freestanding C 用户程序
- 复用当前 staging 模型，不扩展成通用装载器
- 继续沿用现有 trap / handle / seed 机制，不在这一步重构内核 ABI

### 关键任务

- 定义最小用户态编译链：`C -> ELF -> flat binary`
- 让 demo 不再内联手写机器码，而是消费编译产物
- 让用户程序通过 `libsys.h` 调用现有 syscall ABI
- 让 capability seed block 仍位于当前 const 页布局中
- 保持当前 enter-user、write、exit、teardown 的稳定证据链

### 验收结果效果

- 仓库里能直接打开并展示一个真实的用户态 `main.c`
- QEMU 日志能证明该程序确实在 Ring 3 运行并完成输出
- 当前原型不再被描述为“bootstrap 字节码验证桩”，而是“可编译用户程序原型”
- 答辩展示时可以同时展示源码、构建路径和运行结果，叙事质量显著提升

### 完成标志

- 至少 1 个用户程序由 C 编译产生
- 至少 1 个 profile 使用编译产物而不是手写字节数组
- 现有 clean path 仍保有退出与回收证据链

---

## P2：双进程用户态 bring-up

### 方向

在不要求优先级调度先落地的前提下，保留并突出双进程。这个阶段的目标不是“复杂并发模型”，而是证明系统已经具备**两个独立用户进程共同存在并被调度执行**的能力。

### 目标

- 准备两个独立编译的用户程序
- demo 一次创建两个 `ExProcess`
- 每个进程拥有各自独立的 private root 与 seed block
- 两个进程都走自己的创建、运行、退出、回收路径
- 继续使用当前单优先级 RR 调度器

### 推荐展示形态

- `hello`：打印一次启动信息，然后退出
- `counter`：打印若干次带标签的信息，再退出

这两个程序不需要共享对象，不需要跨进程同步，也不需要不同优先级。展示重点是：

- 两个用户程序都是真实编译产物
- 两个程序运行在不同进程上下文中
- 两个程序都能使用自己的 capability / syscall 路径
- 两个程序都能被系统正确回收

### 关键任务

- 为第二个用户程序补齐最小编译目标
- 设计双进程 demo/profile 的启动顺序
- 检查现有 stdout / wait handle seed 机制在双进程下的独立性
- 让串口输出能明显区分两个程序的身份
- 固定双进程 demo 的成功日志锚点

### 验收结果效果

- 串口中能稳定看到两个不同用户程序的输出标签
- 两个进程都成功进入用户态并分别退出
- 一个进程的退出不会破坏另一个进程的地址空间或 handle 状态
- 系统能在 demo 结束后回到稳定 idle，而不是卡死在临时 bring-up 状态
- 展示时可以明确讲出：“这不是单 payload 演示，而是两个独立用户进程”

### 完成标志

- 新增一个双进程 profile，作为 MVP 主演示入口
- 两个进程均为编译型用户程序
- 双进程路径通过日志证明创建、运行、退出、回收完整闭环

## P3：简化版优先级调度

### 方向

这一步不是去做完整调度器重写，而是在当前单优先级 RR 基础上，补一个**足够简单、足够稳定、足够可展示**的优先级层。

### 目标

- 将单 FIFO ready queue 提升为少量固定级别的 ready queues
- 保留当前同优先级 RR 与既有 quantum
- 不引入优先级继承、aging、防饥饿等复杂策略
- 优先用于演示“更高优先级任务先得到 CPU”这件事

### 建议范围

- 优先级等级保持很小，例如 `HIGH / NORMAL / LOW`
- 默认线程优先级保持兼容，避免影响既有大部分路径
- 先做 kernel scheduler 内部能力，再决定是否把不同优先级接入双进程主演示

### 关键任务

- 将 ready queue 改造成小规模多级队列
- 为线程创建路径补默认优先级参数或最小设置接口
- 固定一个 priority smoke profile，验证选择逻辑正确
- 评估是否把双进程 demo 的两个进程设为不同优先级

### 验收结果效果

- 日志能够证明调度器会优先选取高优先级 runnable 线程
- 同优先级线程仍按现有 RR 工作，不引入大范围行为倒退
- README 中“优先级调度”的缺口被部分补齐
- 如果稳定接入双进程主演示，则展示效果会明显更好

### 完成标志

- 至少存在一个简单稳定的 priority regression / demo profile
- 调度器对既有 schedule 路径不产生明显回归
- 若接入主演示，双进程 profile 能稳定体现优先级差异

---

## P4：轻正式化用户态 ABI

### 方向

这一步只做**展示和维护需要的最小收口**，不做大范围重命名，也不把内核内部的 `bootstrap` 术语全部翻新。

### 目标

- 稳定一份用户态可见的 `libsys.h`
- 明确现有 syscall 调用约定
- 尽量让用户程序不再直接写魔法数字
- 仅在确有必要时补 1 到 2 个低成本 syscall

### 建议原则

- 保留内核内部现有实现与命名，避免大重构
- 用户侧先拥有“稳定包装层”，再考虑将来统一正式 ABI
- 若双进程展示只靠 `SYS_WRITE` / `SYS_CLOSE` / `SYS_WAIT_ONE` / `SYS_RAW_EXIT` 即可成立，则不要额外扩 scope
- `SYS_GETPID` 这类 syscall 只有在能显著增强演示辨识度时才进入本阶段

### 关键任务

- 冻结用户态头文件中的 syscall 常量和包装函数
- 记录寄存器约定、返回值约定、错误码约定
- 把用户 demo 源码从“知道太多内核细节”降到“只依赖用户头文件”

### 验收结果效果

- 用户程序源码看起来像“正常 freestanding 用户程序”，而不是 demo hack
- 展示时可以把 ABI 讲成“已有稳定用户侧封装”，而不是“代码里手填 trap 号”
- 后续再做更正式 syscall 分发表时，不需要推翻用户 demo 写法

### 完成标志

- 用户 demo 仅依赖 `crt0.S`、`libsys.h` 和 seed block 公开布局
- 文档中能用一段简洁表述说明当前用户 ABI，而不需要解释内核内部细节

## P5：MVP 演示冻结与文档收口

### 方向

前 4 个阶段解决“能不能做出来”，这一阶段解决“能不能稳定展示、稳定叙述、稳定验收”。

### 目标

- 确定唯一主演示 profile
- 固定构建命令、运行命令、日志锚点
- 把 README / handoff / draft 文档里的原型叙事统一
- 明确 MVP 和 post-MVP 的边界

### 关键任务

- 给双进程 profile 写一段固定演示说明
- 列出演示成功时必须出现的关键日志
- 列出演示失败时最值得观察的日志和排查入口
- 用一页文档解释 RR 基线口径与 `P3` 增强口径之间的关系

### 验收结果效果

- 团队成员可以按一套固定命令稳定复现 MVP
- 答辩时不会陷入“代码有了但故事讲不清”的状态
- 交接文档、README、demo profile 叙事相互一致
- MVP 与后续完整版路线清晰分层，不再相互污染

### 完成标志

- MVP 只有一个官方主入口
- 官方主入口对应固定的成功日志锚点和一句话论证口径
- 团队对“这轮做到了什么、没做到什么”口径统一

## 推荐实施顺序

按“先打前置、再异步推进、最后冻结”的方式，建议顺序如下：

1. `P1` 编译型用户程序
2. `P2 / P3 / P4` 在 `P1` 完成后异步推进
3. `P5` 演示冻结与文档收口

建议把异步推进理解为：

```text
P1 编译型用户程序
        |
        +--> P2 双进程用户态 bring-up
        |
        +--> P3 简化版优先级调度
        |
        +--> P4 轻正式化用户态 ABI
                     |
                     +--> P5 演示冻结与文档收口
```

这样安排的原因是：

- `P1` 是所有后续工作的前提
- `P2` 是这轮展示的核心卖点
- `P3` 团队有余力时值得并行做，但不应反过来阻塞双进程主线
- `P4` 适合与 `P2 / P3` 并行，做成低风险包装层
- `P5` 只在并行工作收敛后进行统一冻结

如果时间继续收紧，最低可交付组合是：

**P1 + P2 + P5**

也就是：

- 有编译型用户程序
- 有双进程
- 有固定主演示入口

这已经足以构成一个正式成立的最小用户态原型。

如果团队空闲兑现，推荐目标则是：

**P1 + P2 + P3 + P5**

也就是在双进程原型成立的基础上，再把简化优先级调度并进去。

## 最终验收口径

### 基线口径

当以下陈述成立时，可以宣布 MVP 达成：

> HimuOS 已实现一个双进程、编译型、能力驱动的用户态原型操作系统。两个用户程序以 C 语言编写并编译产生，在各自独立的进程私有地址空间中进入 Ring 3 执行，通过现有 `int 0x80` syscall 路径获得最小内核服务，并在运行结束后完成各自的 teardown、terminate 与 reaper 回收。当前调度器仍为单优先级 RR，但这不影响该原型在架构上成立，也不影响双进程用户态的演示闭环。

### 增强口径

如果 `P3` 如期完成，则最终答辩口径可以增强为：

> HimuOS 在双进程、编译型、能力驱动的用户态原型基础上，进一步实现了一个简化版优先级调度器。该调度器采用少量固定优先级队列，并在同优先级内保持 RR，从而在不显著增加系统复杂度的前提下提升了展示效果，并部分兑现了 README 对优先级调度的承诺。

---

## 一句话总结

**这轮 MVP 的主线仍然是“把编译型用户程序和双进程真正立起来”，但在团队有余力的情况下，简化版优先级调度值得作为并行增强项加入。** 键盘和 Shell 继续后移，因为它们会明显拉宽战线，而双进程仍然是这次原型从“单样例 bring-up”跃迁为“可展示操作系统”的关键一步。
