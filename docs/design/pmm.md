# KE_PMM 设计与实现文稿

## 概述 & 设计目标

`KE_PMM` 是 `Ke` 层的物理内存管理组件，其职责是在内核早期和后续核心运行阶段，为系统提供统一、稳定的物理页资源管理能力。它只需要关心物理页的可分配、释放、保留和基础统计，不负责完整虚拟内存管理，也不负责用户态地址空间、通用映射策略或更高层的内存对象抽象。当前，`KE_PMM` 能够基于 Loader 交付的物理内存布局建立自身状态，并在 HHDM（高半区直接映射）已存在的前提下，让上层在拿到物理页后可以直接访问其内容。

在结构上，`KE_PMM` 采用了 `DEVICE -> SINK` 的分层思想。`KE_PMM_DEVICE` 作为 Ke 层唯一对外暴露的 PMM 门面（Facade），负责提供统一入口并持有当前后端；`Sink` 负责保存分配状态并实现实际的页框管理逻辑。与平台或启动环境相关的辅助逻辑，例如地址换算、区间裁剪、页对齐和基于 HHDM 的访问辅助，只作为内部 helper 或 adapter 存在。对外接口始终表达“物理页资源”的语义，上层只能提出“我要多少页、是否连续、需要什么限制、归还哪段范围”等需求，不泄露某一种后端算法的内部术语。

> 注意：PMM 当前设计特意忽略了三层设计的 Driver 部分，因为 PMM 的职责边界相对清晰，且内存对于系统本身并不构成外部设备资源；

为了给后续演进留出空间，`KE_PMM` 的基础语义已经与具体后端解耦。这种设计使得在未来需要更强的连续页分配能力或状态追踪时，可以无缝替换更完善的后端，而不影响 Ke 层调用 PMM 的方式，保持了作为长期稳定资源边界的定位。

## 技术选型与后端实现

当前阶段的后端明确选用 `bitmap`（位图），并以此作为唯一的页状态真相源。选择这一方案的理由是它最符合当前阶段的目标：教学上易于说明、验证上易于穷举、后续替换成本可控。与空闲页栈或链表相比，位图天然适合连续页范围分配、范围保留和全局状态审计。

位图后端的具体实现采用了 **每页占用 2 bit** 的结构，一个字节管理 4 个物理页的状态。这种结构非常紧凑且自然地支撑了三态模型：
- `PMM_PAGE_FREE` (0x0)：可分配页
- `PMM_PAGE_ALLOCATED` (0x1)：由 PMM 发出、尚未释放的页
- `PMM_PAGE_RESERVED` (0x2)：静态保留或由系统显式保留的页

分配策略明确采用了“低地址优先、连续优先的 first-fit 扫描”（`BitmapAllocPages`）。也就是说，在满足约束的前提下，优先返回最低物理地址上的首个可用连续区间。这种朴素的扫描便于调试时预测行为，使测试可以直接对分配结果做稳定断言。

## 初始化输入契约

`KE_PMM` 的初始化输入以 Loader 交付的 `BOOT_CAPSULE`（内含 `EFI_MEMORY_MAP`）为准，并通过 `KePmmInitFromBootMemoryMap` 在内部完成受管区间的归一化与裁剪。初始化逻辑显式定义并实现了以下契约：

1. **排序与规整**：输入描述符可视为无序，PMM 内部采用插入排序（Insertion Sort）按物理地址升序排列，并过滤掉 `NumberOfPages == 0` 的空描述符。
2. **严格对齐校验（Fast-Fail）**：如果输入的描述符物理边界非 4KB 对齐，初始化直接返回错误（`EC_NOT_SUPPORTED`）。这是基于防御性编程的考量，若 UEFI 提供非页对齐内存，说明固件或启动数据结构已存在严重问题。
3. **回绕与重叠检查**：显式检查物理范围回绕和区间重叠，发现问题立即拒绝初始化。
4. **受管区间提取**：仅针对类型为可回收（reclaimable）的内存，PMM 计算出所有内存的最低与最高物理地址跨度，作为内部的受管区间（`ManagedBasePhys` 和 `TotalManagedPages`），从而控制位图大小。

## 保留页来源与二次裁剪

PMM 显式维护了一份“启动后仍然保留”的区间来源清单，任何启动后仍被内核继续依赖的区域，都会在 PMM 初始化时通过 `ReserveBootRange` 重新标记为保留。当前实现中至少保留了以下范围：

1. PMM 自身元数据（Bitmap）所占用的页。
2. 低地址 1MB 窗口（legacy low memory），整体保留，不参与普通页分配。
3. `Page 0`（物理地址为 0 的页面），作为 NULL guard 的特例显式保留。
4. 内核镜像（代码段与数据段）。
5. 内核主栈与 IST1 栈。
6. `BOOT_CAPSULE` 自身及其携带的 memory map 缓冲区。
7. 当前仍在使用的页表页。
8. Framebuffer 等启动阶段保留设备区间。

为了保证状态的绝对安全，位图初始化时首先会将整个受管范围全部置为 `RESERVED`，随后仅把属于可回收描述符且未溢出的部分解封为 `FREE`，最后再根据上述内核实体对其覆盖的内存进行二次保留裁剪。

## 元数据自举

由于内核尚未建立完整堆或更高层内存对象系统，PMM 在初始化时自行解决了位图元数据的放置问题：

1. 遍历可回收区间（reclaimable regions），找到第一块容量足以容纳整个位图所需页数、且起点不低于 1MB 的物理连续内存。
2. 直接通过 HHDM 将该物理地址转为虚拟地址，对位图空间进行清零和操作。
3. 这段范围选定后，立即被 `ReserveBootRange` 从可分配集中扣除，并以 `RESERVED` 状态登记。
4. 若没有任何可回收区足以容纳最小元数据，初始化将直接失败，不尝试降级到不完整模式。

## 接口语义与约束

对外暴露的 Ke 层 PMM 接口具有严格的语义约束：

- **`KePmmAllocPages(count, constraints, *outBasePhys)`**：仅能从 `FREE` 状态转为 `ALLOCATED` 状态。`constraints` 支持指定按 2的幂次页对齐（`AlignmentPages`）以及物理地址上限约束（`MaxPhysAddr`）。资源耗尽或超出约束返回 `EC_NOT_ENOUGH_MEMORY`。
- **`KePmmFreePages(basePhys, count)`**：释放连续的物理页，页面状态必须严格为 `ALLOCATED` 才能转换回 `FREE`。传入地址需 4KB 对齐并位于受管范围内，若传入 `RESERVED` 状态或越界的页面则引发 `EC_INVALID_STATE`。
- **`KePmmReservePages(basePhys, count)`**：显式保留页面，状态必须严格为 `FREE` 才能转换为 `RESERVED`。双重释放或非法重叠将被底层安全机制拒绝。
- **`KePmmQueryStats`**：提供 PMM 统计数据：总受管内存、空闲内存、已分配内存、保留内存。

在并发模型上，首版 PMM 暂不引入针对任意中断上下文访问或 Per-CPU Cache 等高级特性。调用方需在合适的位置加入同步机制保障多核心分配的原子性。

## 调试可观测性

目前的 PMM 在设计和实现中已内置了必要的调试可观测性手段：

1. **`KePmmCheckInvariants` 入口**：能够动态遍历整个位图的所有页面，聚合三种状态的数量，并与内部计数的 `FreePages`、`AllocatedPages` 和 `ReservedPages` 进行交叉断言，一旦发现状态腐化（Mismatch）即可返回错误。
2. **关键路径日志（klog）**：在初始化出现回绕、重叠、或找不到元数据存放区时，会精准打印出错内存的段址和长度。
3. **成功摘要输出**：初始化成功后，会在系统日志打印当前 PMM 受管区域起点与终点、总受管页数、各状态页数及位图的存放位置。

## 当前阶段冻结与后置项

通过当前的 `bitmap` 实例和三态模型，HimuOS 已经成功冻结了 PMM 的核心契约与接口。

当前阶段明确后置以下演进方向：
1. 以 `buddy` 为代表的高级连续内存管理后端。
2. Free-page cache 机制或 Per-CPU 局部分配器。
3. 更复杂的内存资源对象（Zone、NUMA 节点识别、DMA 预留池）。
4. 细粒度页状态（如 offline, standby）。

PMM 作为底层基础设施的第一阶段目标已经达成：能稳定、自洽地接管从 Bootloader 交付而来的机器内存状态，并清晰安全地为后续虚拟内存与任务调度组件提供燃料。
